# ADR-0001: AI 最適化開発モデル

- **Status**: Accepted
- **Date**: 2025-06-20
- **Issue**: [#130](https://github.com/mizunowanko-org/vibe-admiral/issues/130)

## Context

vibe-admiral は AI agent が自律的に開発を遂行するシステムである。従来の開発プロセスは人間が読み・書き・承認することを前提に設計されていたが、このシステムでは **95% の読者は AI** である。

従来の開発ドキュメント（要件定義書、仕様書、設計書、進捗報告書）は人間にとって読みやすい形式だが、AI にとってはノイズが多く、コンテキスト抽出の効率が悪い。

## Decision

開発情報を GitHub に集約し、AI がノイズなく正確にコンテキストを抽出できる形式に統一する。

### 情報の配置マッピング

| 開発要素 | 従来の形式 | AI 時代の形式（GitHub 集約） | AI にとっての役割 |
|----------|-----------|----------------------------|-----------------|
| Problem | 要件定義書 | **Issue** | 動機と背景の記録 |
| Spec | 仕様書 / Wiki | **Test** | 実行可能な仕様。嘘をつかない正解の定義 |
| Design | 設計書 / 図面 | **Code** | 具現化された構造 |
| Decision | 口頭 / チャット | **ADR** | 選択の理由、捨てた案、将来への道標 |
| Report | 進捗・完了報告 | **Pull Request** | 変更の意図、議論のプロセス |

### ADR の役割

- AI が迷わないための「ガードレール」
- 「過去にこの構成を試したが却下した」という記録で、AI が同じ過ちを繰り返すことを防ぐ
- Issue / PR と相互リンクし「背景 → 判断 → 実装」を一本の線で理解可能にする

### 事後フィードバックモデル

このモデルの実践的原則として、vibe-admiral は「事後フィードバックモデル」を採用する。

- AI が自律的に調査・計画・実装・テストを実行する
- QA agent が品質を担保し、人間承認は Gate に含めない
- 人間は非同期的に結果を確認し、問題を発見したら新規 Issue で起票する
- 完璧を同期的に求めるより、問題を非同期で回収する方が全体効率が高い

## Consequences

- CLAUDE.md にはリポ固有の開発規約のみを記載し、哲学のエッセンスと ADR へのリンクを含める
- Issue は Problem の記録、Test は Spec の定義、Code は Design の具現、ADR は Decision の記録、PR は Report として機能する
- 人間向けのドキュメント（README.md 等）は存在するが、AI 向けの情報設計を最優先とする
- 今後の設計判断は ADR として記録し、Issue / PR と相互リンクする
