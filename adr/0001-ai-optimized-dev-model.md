# ADR-0001: AI 最適化開発モデル

- **Status**: Accepted
- **Date**: 2025-06-01
- **Issue**: [#130](https://github.com/mizunowanko-org/vibe-admiral/issues/130)

## Context

vibe-admiral は Claude Code を使った並列開発を統括するシステムである。このシステムでは、コードを読み・書き・レビューする主体の大半が AI（Claude Code セッション）であり、人間は方向性の決定と最終的な受け入れテストに集中する。

従来の開発ドキュメント体系（要件定義書、仕様書、設計書、議事録、進捗報告書）は人間が読むことを前提に設計されていた。しかし AI が開発の中心を担う環境では、以下の問題が発生する:

1. **散在する情報源**: 仕様が Wiki、設計が Confluence、議事録が Slack に分散し、AI がコンテキストを収集するコストが高い
2. **形式知の欠落**: 口頭やチャットで行われた意思決定が記録されず、AI が同じ失敗を繰り返す
3. **実行不可能な仕様**: 自然言語の仕様書は解釈のブレが避けられず、AI にとって曖昧な指示になる
4. **同期的な人間承認のボトルネック**: 並列 sortie のスループットが人間の応答速度に律速される

## Decision

**95%の読者は AI である** という前提に基づき、開発情報を GitHub に集約する。各開発要素を以下の形式で表現する:

| 開発要素 | 従来の形式 | AI 時代の形式（GitHub 集約） | AI にとっての役割 |
|----------|-----------|---------------------------|-----------------|
| Problem | 要件定義書 | **Issue** | 動機と背景の記録 |
| Spec | 仕様書 / Wiki | **Test** | 実行可能な仕様。嘘をつかない正解の定義 |
| Design | 設計書 / 図面 | **Code** | 具現化された構造 |
| Decision | 口頭 / チャット | **ADR** | 選択の理由、捨てた案、将来への道標 |
| Report | 進捗・完了報告 | **Pull Request** | 変更の意図、議論のプロセス |

### ADR の役割

ADR は AI が迷わないための「ガードレール」である。

- 「過去にこの構成を試したが却下した」という記録で、AI の同じ過ちの繰り返しを防ぐ
- Issue/PR と相互リンクし「背景→判断→実装」を一本の線で理解可能にする
- CLAUDE.md にはエッセンスのみ記載し、詳細な経緯は ADR に委ねることで、AI のコンテキストウィンドウを節約する

### 事後フィードバックモデルとの関係

この ADR で定義する「AI 最適化開発モデル」は情報設計の哲学であり、「事後フィードバックモデル」はその哲学に基づく運用モデルである。

- **AI 最適化開発モデル**: 情報を AI が最も効率的に消費できる形式で配置する
- **事後フィードバックモデル**: AI が自律的に品質を担保し、人間は事後に問題を回収する

## Consequences

### Positive

- AI が Issue → Code → Test → PR → ADR の一本道でコンテキストを取得でき、調査コストが減る
- ADR により過去の判断を参照できるため、AI が却下済みの案を再提案するリスクが減る
- Test が仕様の正解定義として機能し、AI の実装が仕様に一致しているかを自動検証できる

### Negative

- ADR の記述コスト — 判断のたびに ADR を書く必要がある（ただし AI が下書きを生成できる）
- GitHub に集約するため、GitHub 外のツール（Notion, Confluence 等）との併用は情報の分散を招く

### Risks

- ADR が陳腐化するリスク — コードが変わっても ADR が更新されない可能性がある。PR レビュー時に関連 ADR の確認を促す運用でカバーする
